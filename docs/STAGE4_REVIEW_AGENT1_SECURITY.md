## Narrative â€” strongest mitigations (with evidence)

1. **The "free Pro / infinite credits" holes are genuinely closed at the database layer, and the tests prove it.** This was the headline P0-3/P0-7a risk. The built posture is exactly right: entitlement and ledger tables are **client-read-only** (0002: SELECT-own policy, no write policy, `REVOKE ALL`/`GRANT SELECT`), all other learner tables are **deny-by-default per-table** (0003: `ENABLE`+`FORCE` RLS + own-row `USING/WITH CHECK`), and the *only* write path is two `SECURITY DEFINER` functions that are **service_role-EXECUTE-only with a pinned `search_path`**. Crucially, the negative tests are real: `test_rls_entitlement.py` and `test_credit_ledger.py`/`test_entitlement_events.py` assert that an `authenticated` session is *denied* the self-grant, the self-mint, the direct fn EXECUTE, and the cross-user read/write â€” and then independently confirm nothing was mutated. The isolation test even **derives the table list from the schema** so a future un-RLS'd table fails by construction. This is the difference between "we wrote RLS" and "we proved RLS denies the attack."

2. **The credit ledger is a correct double-entry, fail-closed-at-zero, idempotent, concurrency-safe design â€” with a true two-thread proof.** `post_credit_entry` locks the user row `FOR UPDATE`, recomputes the authoritative balance as a signed sum (order-independent), refuses sub-zero spend without ever clamping, validates refunds against the referenced spend and the remaining-refundable amount, and is idempotent on `client_event_id` with an ON-CONFLICT fallback for the SELECTâ†’INSERT race. `test_credit_ledger.py:187-228` spins two real threads through a barrier and asserts exactly one of two interleaved full-balance spends wins and the balance never goes negative. That concurrency test is the kind of thing most teams skip.

3. **Payment verification crypto is correct** (constant-time HMAC with a folded length difference and no early return; replay window via timestamp tolerance; raw-body signing) **and the entitlement transition is exactly-once with out-of-order safety** (event dedupe table, `GREATEST` extend, fail-closed on unknown kind/user, rolled back so a bad event is never marked processed). The *logic* is sound and well-tested.

4. **Fail-closed is the consistent default across every seam**, and it's the right bias for a minors-in-scope product: unconfigured AI relay throws (`UnconfiguredAiRelay`); Gemini with no key never touches the transport; moderation denies on block/timeout/unknown with no default-allow branch; cost guard denies on null/negative/over-cap and when a spend provider throws; grant guard denies on unknown counts and unavailable verifiers; media authz denies `proAudio` to non-Pro outright and refuses over-broad signing requests as *unconstructable*. The `RelayText` untrusted-output box and the `AnonymousClaimToken` server-only type both make the unsafe path *unrepresentable at the type level*, which is stronger than a runtime check.

5. **Secret hygiene is enforced, not just asserted.** `test_secret_scan.py` is a real CI tripwire with full-shape patterns, planted-secret fixtures, near-miss negative tests, and a clean-repo assertion; `service_role_contract.dart` version-controls the read-from-env invariant and the rotation runbook.

6. **The analytics PII guard is built and fails closed** (`AllowListAnalytics` + closed taxonomy + forbidden-token + PII-value detection, dropping violations in release). This is the one substantive security/privacy control that is fully built end-to-end on the client, and it's wrapped from day one.

## Narrative â€” the real residual risks

The built logic is strong; the residual risk is almost entirely at the **edges that aren't built yet** plus a few owner decisions:

- **The 4 Deno Edge hosts are the unbuilt perimeter that holds the service-role key (A-E2, P0).** Every database-layer mitigation above is predicated on the service-role key staying server-side and the hosts composing the seams *in the specified order, verify-before-mint, with a pinned upstream endpoint*. The composition is correct on paper, but no host code exists to review. This is the single biggest reason go-live needs a *second* security pass focused on the hosts (SSRF in the relay proxy, key-in-logs, client-asserted entitlement, missing verify-before-mint).

- **Payment provider mismatch (A-T8, P1).** The locked web vendor is **Razorpay**, but the verifier implements only the **Stripe** `t=â€¦,v1=â€¦` scheme. As built, wiring Razorpay would fail-closed-reject all real events (no Pro grants) â€” or, worse, invite a hand-rolled bypass. The reusable parts (HMAC primitive, `PaymentEvent` normalization, `apply_entitlement_event`) are fine; the provider-specific verify path is missing for the actual vendor. Play Billing needs S2S receipt validation, not an HMAC, at all.

- **Per-user cost-cap bypass on moderation-blocked outputs (A-D2, P1).** Because spend is recorded only after a *successful* `inner.complete()` and output moderation lives inside the inner relay, an attacker can burn paid Gemini round-trips that trip output moderation without depleting their per-user daily cap. The global ceiling still bounds total spend, but the per-user economic control has a hole for the blocked/failed-output class.

- **Audit durability is plumbed but not durable (A-R1, P1).** `GrantAuditSink`/`ModerationAuditSink` call sites exist and route every deny/verdict, but the local defaults are no-ops. Non-repudiation of security actions depends on a durable, append-only store that is a go-live dependency.

- **Owner decisions still open:** real cost caps + per-model prices (A-D3), velocity-cap values (A-E3), the durable audit store (A-R1), and JWT issuance/rotation/revocation + bot-defense on the auth path (A-S1/A-S3) â€” all design-only or placeholder today.

None of these contradicts the build-ahead's own framing ("production is mostly wiring credentials, not writing logic"). The logic largely lives up to that claim. The caveat a rigorous reviewer must add is: **"mostly wiring" still includes the service-role-key-bearing hosts and a payment-provider verify path that is currently mismatched to the chosen vendor â€” those are not trivial wiring, and they must be reviewed as security-critical before the backend is provisioned.**
